Building one 360° point cloud from 10 stereo shots

The robot spins in place and takes 10 stereo photos. It has no compass and no wheel encoders, so it doesn't know how far it turned. This page shows how the code figures out each shot's angle from the images and merges all 10 into one cloud.

1 The 10 shots — a ring of views

Robot sits in one spot, rotates ~36–40° between each shot. Camera field of view is ~90°, so neighbours overlap a lot (they see the same wall from a slightly different angle). The last shot wraps around to overlap the first.

Each red dot = one camera position (shot 0…9). The shaded wedge = what that camera sees. Notice neighbours share a big slice — that shared slice is what lets us match them.

Hover a shot to see its ~90° view cone and which neighbours it overlaps.

camera / shot field of view

2 Where do the “45 pairs” come from?

To find which shots overlap, the code compares every shot with every other shot. It doesn't know in advance which ones line up, so it tries all combinations.

Hover any shot → it lights up its 9 possible partners.

It's the handshake problem: 10 people in a room, how many unique handshakes? Each of the 10 shots can pair with the other 9, but pair (3,7) is the same as (7,3), so divide by 2:

C(10,2) = 10 × 9 / 2 = 45

So the code runs the matching routine 45 times, once per unique pair. Most pairs don't overlap (e.g. shot 0 vs shot 5 face opposite walls) → rejected. Only the pairs that truly share a view survive — about 11 of them here.

neighbour pair (overlaps, kept) tried, usually rejected

The same 45, as a table

Every cell above the diagonal is one pair. Count them → 45. The diagonal neighbours (0-1, 1-2, …) plus the wrap (0-9) are the ones that reliably overlap.

3 Turn each shot's depth map into 3D points

Before matching, every shot becomes a little 3D cloud in its own camera frame. The depth camera gives a distance Z for each pixel; the pinhole formula spreads pixels out into 3D.

# pixel (u,v) + depth Z → 3D point Z = depth(u, v) X = (u − ppx) · Z / fx Y = (v − ppy) · Z / fy

fx, fy = focal length, ppx, ppy = image centre (from intrinsics.txt). A pixel near the image centre with depth 0.5 m lands almost straight ahead; a pixel near the edge lands off to the side. Function: back_project_dense().

4 Find & match landmarks between two shots

For a pair that overlaps, find the same physical points in both shots.

We match on the raw IR photo — not the depth map.

Features are detected and matched on ir_left.png (the raw infrared image). The depth map depth.npy is used only to attach a 3D coordinate to each matched feature. ir_right.png is not used here — it was already spent earlier to compute the depth.

The sub-steps

a. Brighten the IR (CLAHE) image

IR is dim (avg ~55/255); the detector found ~30 corners. CLAHE boosts local contrast → ~30× more. Detection-only — the original IR still colours the cloud.

b. Detect corners (ORB) image

~3000 corners/image, each with a 256-bit binary descriptor (a fingerprint of its texture patch). SIFT is a stronger fallback when ORB is weak on a pair.

c. Lift to 3D depth used HERE only

Each corner reads depth in a 5×5 window → median (noise-robust), reject if zero or the window varies >20 cm. Back-project the survivors to a 3D point. This is the only place depth enters step 4.

d. Match + clean image

Lowe 0.75: best match must clearly beat the 2nd-best (drop ambiguous). Mutual: A and B must pick each other. Then a fundamental-matrix RANSAC on the 2D image coords removes geometrically-impossible matches (red below).

good match (kept) false match (removed by Lowe + fundamental-matrix RANSAC)

Each surviving green match is the same point in shot A and shot B. Step c gave it a 3D position in each → the match becomes a pair of 3D points, the input to Kabsch.

Why match on IR, not the depth map? Depth maps are smooth, noisy and full of holes — no fine texture for descriptors to lock onto. Raw IR has real texture (edges, patterns) → distinctive, repeatable features. Standard RGB-D practice: match on appearance, measure geometry from depth.

🔍 Concepts behind step 4

Three terms from step 4, explained plainly — with where each is used.

1 · CLAHE used in step 4a

Contrast Limited Adaptive Histogram Equalization — a contrast booster.

A histogram counts how many pixels have each brightness (0=black … 255=white). The IR is dim, so all the counts bunch into a narrow dark band — faint texture is there but invisible.

  • Equalization — stretch/spread the brightnesses across the full 0–255 range → contrast up, faint texture becomes visible.
  • Adaptive — done separately on each small 8×8 tile, not the whole image → fixes uneven lighting (one bright corner won't wash out the rest).
  • Contrast Limited — cap (clip) the tallest histogram bars first (clipLimit=3.0) so flat areas don't blow up into noise.

Result: dim shots went from ~30 detectable corners to ~900. Code: cv2.createCLAHE(clipLimit=3.0, tileGridSize=(8,8)) in _detect().

brightness histogram: bunched (dim) → spread (clear)

2 · ORB used in step 4b

Oriented FAST + Rotated BRIEF — a fast, free corner detector + describer.

FAST: ring of 16 pixels around a candidate

ORB does two jobs:

  • FAST = the detector. For a candidate pixel, look at the ring of 16 pixels around it. If ≥9 in a row are all brighter (or all darker) than the centre by a threshold → it's a corner. Only comparisons → very fast.
  • BRIEF = the descriptor (next concept) — turns each corner's patch into a binary fingerprint.
  • ORB extras — gives each corner an orientation (so the fingerprint is rotation-invariant, vital for our ~40° spin) and runs on a scale pyramid.

Output per corner: (x, y, angle, scale) + a 256-bit descriptor. ~3000/image. Free + light → great on the Pi. Code: cv2.ORB_create(nfeatures=3000).

3 · The descriptor (fingerprint) built in 4b, compared in 4d

How a patch of pixels becomes a number you can compare.

ORB uses BRIEF. Around the corner, a fixed set of 256 point-pairs is sampled inside the patch. For each pair (a, b), one test:

intensity(a) < intensity(b) ? bit = 1 : bit = 0

256 tests → a 256-bit string = the fingerprint. The patch is first rotated to the corner's orientation, so the same physical corner gives the same bits even after the camera turned.

Comparing two fingerprints = Hamming distance: XOR them, count the differing bits. Small distance → same patch → candidate match. A CPU does this in nanoseconds, so matching 3000×3000 is cheap.

SIFT (our fallback) instead builds a 128-number vector from gradient-orientation histograms, compared by ordinary (L2) distance — heavier but more robust. Same idea: patch → comparable number.

256 pixel-pair tests → 256 bits; XOR two → Hamming distance

5 Solve the rotation between the pair (Kabsch + RANSAC)

We have the same points in two clouds. Find the single rotation + shift that lays one cloud exactly onto the other — that is the camera's turn between the two shots.

# matched 3D points P (in A) and Q (in B) H = Pcᵀ · Qc # cross-covariance U,S,Vᵀ = SVD(H) R = V · diag(1,1,±1) · Uᵀ # rotation t = mean(Q) − R · mean(P) # shift

This is the Kabsch algorithm — a closed-form best-fit rotation. Wrapped in RANSAC: pick 3 random matches, fit, count how many other matches agree (within 6 cm), keep the best fit, refit on all agreers. RANSAC throws out any wrong match that slipped through. Functions: kabsch(), kabsch_ransac().

Output per pair = a transform T (rotation R + shift t). No angle was assumed — it's measured from the points.

6 Make all 10 agree (pose graph + loop closure)

Chaining the pair-transforms 0→1→2→…→9 builds up small errors (drift): by the time you reach shot 9 it doesn't line up with shot 0, leaving a gap. The wrap pair (9,0) says “these two actually overlap” — that contradiction reveals the drift, and we spread the correction over all 10 shots.

Nodes = the 10 shots. Edges = the measured pair-transforms from step 5:

  • Green = neighbour edges (the chain, trusted).
  • Yellow = the loop-closure edge 9→0.

Open3D's global_optimization (Levenberg–Marquardt) nudges all 10 poses at once so every edge is satisfied as well as possible → the gap closes and the ring is consistent. Function: optimize_poses().

Press optimize on the left to watch the gap close.

7 Stack, clean, colour → the final cloud

Now each shot has a correct pose in one shared frame. Re-project every shot's full depth, move it by its pose, and pile them together.

close mid far
  • Stack all 10 transformed clouds into one.
  • Voxel downsample (5 mm grid) — the heavy overlap means many duplicate points; keep one per cell.
  • Outlier removal — drop floating specks (passive-stereo noise).
  • Colour by distance — red = close, blue = far.

The top-down view is a donut: an empty hole in the middle (where the robot stood — it can't see itself) ringed by the room. The clean hole proves all 10 poses converge on one centre = the alignment is correct. Functions: shot_world_cloud(), finalize().

The real output is pointcloud_360.ply + pointcloud_360_preview.png in this folder.
